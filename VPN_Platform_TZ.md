# Техническое задание
## Корпоративная платформа управления VPN-инфраструктурой

---

## 1. Общие положения

### 1.1. Назначение документа

Настоящее Техническое задание (ТЗ) определяет требования к разработке, внедрению и эксплуатации внутренней корпоративной платформы управления VPN-инфраструктурой (далее — «Система»).

### 1.2. Цели проекта

- Предоставление сотрудникам корпоративного доступа к внешним интернет-ресурсам через управляемую VPN-инфраструктуру.
- Централизованное управление VPN-нодами, пользователями и подписками.
- Автоматизация деплоя, обновления конфигураций и отказоустойчивости без участия конечного пользователя.
- Полный аудит действий пользователей и администраторов.
- Интеграция с корпоративным SSO и отсутствие анонимности доступа.

### 1.3. Границы проекта

- **Включается:** разработка портала самообслуживания, backend, промежуточного сервера, системы управления нодами, автоматический деплой VPN-стека, подписочная модель, SSO-интеграция, аудит, мониторинг.
- **Исключается:** биллинг, оплата, тарификация, платёжные системы, публичная коммерческая продажа доступа.

### 1.4. Сокращения и термины

| Термин | Определение |
|--------|-------------|
| **Нода** | Виртуальная машина (VM), выступающая выходной точкой VPN-трафика. |
| **Подписка (subscription)** | Подписочная ссылка, предоставляющая клиентскому приложению актуальные настройки подключения. |
| **Промежуточный сервер** | Внутренний сервер в РФ, принимающий трафик сотрудников, фильтрующий и перенаправляющий его к нодам. |
| **SSO** | Single Sign-On — корпоративная система единого входа. |
| **Кабинет** | Личный кабинет сотрудника в веб-интерфейсе. |

### 2.2. Компоненты

| Компонент | Расположение | Назначение |
|-----------|--------------|------------|
| **Фронтенд** | Европа | Веб-интерфейс для сотрудников и администраторов, отображение данных, взаимодействие с LLM. |
| **Бэкенд + БД** | Россия | Хранение данных, управление доступом, обработка заявок, генерация подписок, аудит. |
| **Промежуточный сервер** | Россия | Приём корпоративного трафика, фильтрация, применение политик, перенаправление через защищённый канал. |
| **Ноды (ферма)** | Европа | Выходные точки VPN, маскировка трафика под HTTPS, балансировка, автоматическое переключение при блокировке. |

### 2.3. Защищённый канал

- Между российским бэкендом и европейским фронтендом — защищённый канал (VPN / TLS / IPsec по усмотрению инфраструктурной команды).
- Между промежуточным сервером и нодами — трафик перенаправляется через тот же защищённый канал или выделенный туннель.

## 3. Функциональные требования

### 3.1. Аутентификация и авторизация (SSO)

- **FR-1.1.** Система должна поддерживать вход через корпоративный SSO (SAML 2.0 / OAuth 2.0 / OpenID Connect).
- **FR-1.2.** После аутентификации пользователь попадает в личный кабинет.
- **FR-1.3.** Идентификация пользователя точная — анонимный доступ невозможен. В системе фиксируется корпоративный UPN/email/ID.
- **FR-1.4.** Ролевая модель:
  - `Сотрудник` — может запросить доступ, получить подписку, просмотреть свои заявки.
  - `Администратор` — управляет нодами, пользователями, заявками, политиками, просматривает аудит.
  - `Супер-администратор` — полный доступ, включая настройки системы.

### 3.2. Жизненный цикл заявки на доступ

- **FR-2.1.** Сотрудник в личном кабинете нажимает «Получить доступ к VPN».
- **FR-2.2.** Система автоматически создаёт заявку со статусом `pending`.
- **FR-2.3.** Заявка проходит обработку (автоматическую или с этапом согласования — конфигурируется администратором).
- **FR-2.4.** После одобрения система генерирует подписочную ссылку (subscription URL) и QR-код.
- **FR-2.5.** Сотрудник получает ссылку и QR-код в интерфейсе и может импортировать их в клиентское приложение.
- **FR-2.6.** Зависшие заявки не теряются — система автоматически добивает их при перезапуске.
- **FR-2.7.** При ошибках выдачи пользователю показывается понятное сообщение, а ошибка записывается для разбора администратором.

### 3.3. Подписочная модель (динамические конфигурации)

- **FR-3.1.** Пользователь получает не статичный конфиг, а подписочную ссылку (`https://<domain>/api/sub/<token>`).
- **FR-3.2.** Клиентское приложение регулярно (периодичность настраивается, по умолчанию каждые 5 минут) запрашивает актуальные настройки по подписочной ссылке.
- **FR-3.3.** Динамическое обновление:
  - Новая нода добавлена — автоматически появляется в подписке.
  - Нода заблокирована — выпадает из подписки, клиент переключается на доступную.
  - Смена IP или домена — обновляется при следующем запросе подписки.
- **FR-3.4.** Ручная перенастройка клиентского приложения не требуется.
- **FR-3.5.** Подписка содержит конфигурации для нескольких протоколов (sing-box, xray, hysteria2, WireGuard и др.) в формате, совместимом с популярными клиентами.

### 3.4. Управление нодами и оркестрация

- **FR-4.1.** Система хранит реестр VM (нод): IP, домен, регион, облако, статус, протоколы, нагрузка.
- **FR-4.2.** Автоматический деплой VPN-сервера на новую VM одной кнопкой (аналог Amnezia/Outline Manager):
  - Администратор добавляет VM в панель (IP, SSH-ключи, ОС).
  - Система через Ansible/Terraform/SSH разворачивает Docker-контейнеры с VPN-стеком (sing-box, xray, hysteria2, wg и др.).
  - После деплоя нода автоматически регистрируется в системе и становится доступной для подписок.
- **FR-4.3.** Обновление конфигурации нод централизованно — без участия пользователей.
- **FR-4.4.** Добавление новой ноды происходит прозрачно, без влияния на текущих пользователей.
- **FR-4.5.** Мониторинг состояния нод: health-check, загрузка, доступность, блокировка.
- **FR-4.6.** Автоматическое переключение клиентов на работающую ноду при недоступности текущей.
- **FR-4.7.** Необходимо 3–4 VM из разных облаков/дата-центров для резервирования.

### 3.5. Трафик и маскировка

- **FR-5.1.** Весь трафик сотрудников направляется на промежуточный сервер внутри страны.
- **FR-5.2.** Промежуточный сервер перенаправляет трафик через защищённый канал к европейским нодам.
- **FR-5.3.** Трафик маскируется под обычный HTTPS — для внешнего наблюдателя соединение выглядит как легитимное.
- **FR-5.4.** Несколько нод = несколько выходных IP. При блокировке одной — автоматическое переключение на другую.

### 3.6. Контроль и безопасность

- **FR-6.1.** Весь трафик проходит через промежуточный сервер — возможна фильтрация, логирование, применение политик.
- **FR-6.2.** Возможность отзыва доступа:
  - Полное удаление подписки.
  - Ограничение срока действия (TTL).
  - Ограничение объёма трафика.
- **FR-6.3.** Администратор может вручную заблокировать пользователя или приостановить его подписку.
- **FR-6.4.** Поддержка белых/чёрных списков ресурсов (опционально, через промежуточный сервер).

### 3.7. Аудит и логирование

- **FR-7.1.** Система ведёт аудит всех операций: выдача доступа, ошибки, отзыв, создание/удаление нод, изменение политик.
- **FR-7.2.** Логи содержат: timestamp, ID пользователя, тип операции, результат, IP-адрес, детали.
- **FR-7.3.** Администратор может просматривать аудит в веб-интерфейсе с фильтрацией и поиском.
- **FR-7.4.** Логи хранятся не менее 90 дней (настраивается).

### 3.8. Отказоустойчивость

- **FR-8.1.** Зависшие заявки не теряются — система их добивает при перезапуске.
- **FR-8.2.** Ошибки при выдаче сопровождаются понятным сообщением пользователю и записью для разбора.
- **FR-8.3.** Новые ноды добавляются прозрачно, без влияния на текущих пользователей.
- **FR-8.4.** При падении ноды клиенты автоматически переключаются на резервную.
- **FR-8.5.** Бэкенд и БД развёрнуты с репликацией/бэкапированием (по усмотрению инфраструктурной команды).

## 4. Нефункциональные требования

### 4.1. Производительность

- **NFR-1.1.** Время отклика API не более 500 мс для 95-го перцентиля.
- **NFR-1.2.** Поддержка одновременной работы не менее 500 пользователей (настраиваемый лимит).
- **NFR-1.3.** Генерация подписки и QR-кода — не более 3 секунд.

### 4.2. Надёжность

- **NFR-2.1.** Доступность бэкенда — 99.9%.
- **NFR-2.2.** Регулярное резервное копирование БД (ежедневно, с хранением 30 дней).
- **NFR-2.3.** Автоматический restart сервисов при падении (systemd / Docker / k8s).

### 4.3. Безопасность

- **NFR-3.1.** Все API-вызовы только по HTTPS/TLS 1.3.
- **NFR-3.2.** Подписочные ссылки содержат токен с достаточной энтропией (минимум 128 бит), срок действия ограничен.
- **NFR-3.3.** Хранение паролей и ключей — только в зашифрованном виде (HashiCorp Vault, AWS KMS, или аналог).
- **NFR-3.4.** Изоляция контейнеров на нодах (Docker, rootless по возможности).
- **NFR-3.5.** Регулярное обновление VPN-стека и ОС на нодах.

### 4.4. Масштабируемость

- **NFR-4.1.** Возможность добавления новых нод без остановки системы.
- **NFR-4.2.** Возможность горизонтального масштабирования бэкенда (stateless-архитектура).
- **NFR-4.3.** Поддержка увеличения числа пользователей в 10 раз без изменения архитектуры.

### 4.5. Совместимость

- **NFR-5.1.** Подписки совместимы с клиентами: sing-box, v2rayN, Nekoray, Hiddify, Amnezia, WireGuard и др.
- **NFR-5.2.** Веб-интерфейс корректно работает в последних версиях Chrome, Firefox, Edge, Safari.

## 5. Интерфейсы и взаимодействие

### 5.1. Веб-интерфейс (фронтенд)

- Страница входа через SSO.
- Личный кабинет сотрудника:
  - Статус текущей подписки (активна / истекает / отозвана).
  - Кнопка «Получить доступ» / «Продлить доступ».
  - Отображение подписочной ссылки и QR-кода.
  - Инструкции по импорту в клиентские приложения.
- Панель администратора:
  - Управление нодами (список, добавление, удаление, статус, health-check).
  - Управление пользователями (поиск, блокировка, отзыв подписки, ограничение трафика).
  - Управление заявками (очередь, одобрение, отклонение).
  - Аудит-лог (фильтры, поиск, экспорт).
  - Настройки политик и лимитов.

### 5.2. API (бэкенд)

- REST API / JSON для фронтенда.
- API для клиентских приложений:
  - `GET /api/sub/<token>` — получение актуальной конфигурации подписки.
- API для промежуточного сервера (при необходимости):
  - Синхронизация политик, списка активных пользователей.
- API для нод (опционально, для health-check и метрик).

### 5.3. Интеграции

- Корпоративный SSO (SAML / OAuth2 / OIDC).
- Система мониторинга (Prometheus + Grafana — опционально, по усмотрению).
- Система уведомлений (Email / корпоративный мессенджер — опционально).
- Система оркестрации (Ansible / Terraform для деплоя нод).

## 8. Критерии приёмки

- [ ] Пользователь может войти через SSO, запросить доступ, получить подписку и QR-код.
- [ ] Подписка динамически обновляется при изменении списка нод без ручной перенастройки клиента.
- [ ] Администратор может добавить новую VM, нажать кнопку деплоя, и нода автоматически становится доступной.
- [ ] При блокировке ноды клиент автоматически переключается на резервную.
- [ ] Все операции (выдача, отзыв, ошибки) фиксируются в аудите.
- [ ] Доступ можно отозвать полностью или ограничить по сроку/трафику.
- [ ] Система выдерживает нагрузку 500+ одновременных пользователей.
- [ ] Все компоненты развёрнуты в соответствии с географическим распределением (фронтенд/ноды — Европа, бэкенд/промежуточный сервер — Россия).

## 9. Приложения

### Приложение А. VPN-стек для деплоя на ноды

Рекомендуемый набор (настраивается администратором):
- **sing-box** — универсальный прокси/VPN-клиент.
- **xray** — продвинутый прокси-фреймворк (VLESS, VMess, Trojan).
- **hysteria2** — QUIC-based протокол с обфускацией.
- **WireGuard** — простой и быстрый VPN-протокол.
- **Docker + Docker Compose** — контейнеризация и изоляция.

### Приложение Б. Формат подписки

- Поддержка форматов: **sing-box JSON**, **Base64-субскрипция (v2ray)**, **Clash YAML** (опционально).
- Подписочная ссылка — `https://<domain>/api/sub/<token>`.
- Ответ — актуальный JSON/YAML с актуальным списком нод, протоколов и параметров.
